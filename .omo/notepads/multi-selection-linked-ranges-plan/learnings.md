- Task 1: Added linked-mode public model types in `src/types.ts` without deleting or replacing legacy `SelectionRange`. `SelectionProps` now accepts optional linked props (`selectionId`, `linkedMode`, `linkedData`, linked callbacks) while legacy callers can continue omitting linked mode and using existing `ranges`/callbacks. `src/index.ts` re-exports every new linked public type.
- Task 2: Added `pixelRectsToPercentRects`, `percentRectsToPixelRects`, and `percentRectListsEqual` in `src/geometry.ts`, exported via `src/index.ts`. Linked controlled geometry remains typed as `Record<string, PercentOverlayRect[]>`; pixel `OverlayRect[]` is only for transient overlay/hit-test/handle/popover geometry. `npm run typecheck` passes.
- Task 3: Updated `src/Selection.tsx` so legacy mode still recomputes persisted SVG rects from `ranges`, while linked mode validates non-empty `selectionId`, filters `linkedData.items` by endpoint or `rectsBySelectionId[selectionId]`, converts only that selection's percent rects to transient pixel rects, and renders rects with `data-range-id` plus `data-selection-id`. Linked selected styling now reads `linkedData.selectedRangeId`; legacy selected styling still reads `selectedRangeId`. `npm run typecheck` passes.
- Task 4: Added `src/linkedRegistry.ts` and exported its helpers from `src/index.ts`. Linked `Selection` components now register their container by normalized `selectionId`, unregister on cleanup, ignore duplicate ids after a development warning, expose DOM-ordered registered containers, resolve native node/offset endpoints to local plain-text offsets, and keep `LinkedSelectionData.selectionOrder` synchronized through `onLinkedDataChange`. `npm run typecheck` passes and output is captured in `.omo/evidence/task-4-multi-selection-linked-ranges-plan.md`.
- Task 5: `useTextSelection` now has an optional linked path that resolves endpoints through `linkedRegistry`, clips the native Range to each registered container it intersects, persists only percent rects, and exposes one `LinkedSelectionRange` for confirmation. `Selection` linked mode appends exactly one linked item and calls linked callbacks, while same-container linked selections remain linked-shaped and may also fire representable legacy callbacks. `npm run typecheck` passes; LSP diagnostics tool was unavailable in-session.
- Task 6: Upgraded `src/Selection.tsx` for linked-mode handle behavior. `onLinkedUpdateRange` is now wired via a bridging ref (`onLinkedUpdateRangeRef`), and `linkedSelectionIdRef` lets the `[]`-dep `startHandleDrag` callback read the current linked selection id. `startHandleDrag` sets the drag anchor from the linked item's local offset for same-Selection items (both endpoints equal current `selectionId`) and sets anchor to `-1` for cross-Selection items, blocking their drag in pointermove. The pointermove effect has a linked branch that computes new `lo`/`hi` offsets, rebuilds the DOM Range, converts resulting pixel rects to percent rects via `pixelRectsToPercentRects`, updates the `LinkedSelectionRange` and `LinkedSelectionData`, and calls `onLinkedUpdateRange` + `onLinkedDataChange` — never `onUpdateRange`. The persisted handle rendering condition changed from `!linkedContext` to an inline same-Selection check: cross-Selection linked items have no handles. Hit testing (`handleContainerClick`) and popover anchoring already worked in linked mode because they use `persistedRects` and `currentSelectedRangeId` which are linked-aware. `npm run typecheck`, `npm run build`, and `npm run lint` all pass.
